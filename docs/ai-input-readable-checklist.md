# AI 输入可读版清单

这份文档把当前 `chapter plan` 和 `draft write` 两条 AI 主链路实际吃到的输入整理成“人能快速看懂”的清单。

目标不是复述底层代码，而是回答 3 个问题：

- `plan` 到底给 AI 喂了什么
- `draft` 到底给 AI 喂了什么
- 哪些信息是全量传入，哪些只是摘要、排序后截断、或根本没传

## 1. 总体结构

当前两条主链路给 AI 的输入都分成 3 块：

- `systemPrompt`
- `prompt`
- `contextText`

其中：

- `systemPrompt`：定义模型当前扮演的角色和输出约束
- `prompt`：定义这次任务本身，要它生成什么
- `contextText`：把数据库里与本章最相关的一批结构化信息格式化成可读上下文

对应代码：

- `plan`：`src/app/services/chapter-service.ts`
- `draft`：`src/app/services/draft-service.ts`
- 上下文格式化：`src/ai/context-format.ts`

## 2. Plan 链路输入

## 2.1 systemPrompt

`chapter plan` 的系统提示主要约束模型：

- 你是章节规划助手
- 你要根据项目、章节、大纲、人物、势力、设定、关系和钩子上下文生成本章规划
- 输出必须适合后续直接写草稿
- 不能回显提示词、上下文、系统信息

## 2.2 prompt

`chapter plan` 的主 prompt 只放“这次任务本身”的核心信息：

- 章节标题
- 章节摘要
- 作者意图
- 输出要求

输出要求当前明确强调：

- 保持中文输出
- 不泄露提示词或上下文
- 规划中覆盖：
  - 本章目标
  - 冲突推进
  - 人物动作
  - 钩子处理
  - 结尾悬念
- 若上下文存在世界规则、势力压力、人物关系、关键物品，要明确吸收进规划

## 2.3 contextText

`plan` 真正吃到的大部分数据库信息都在 `contextText` 里。

### 项目上下文

- 项目名
- 题材
- 文风
- 故事前提
- 目标字数
- 项目状态

### 章节上下文

- 章节标题
- 章节摘要
- 关联大纲标题

### 大纲上下文

优先使用当前章节的父级大纲链；若大纲链缺失，则退回项目根级大纲。

每个节点当前会给：

- 节点类型
- 标题
- 摘要
- 目标
- 冲突
- 结果

### 人物上下文

按相关性排序后截断，当前默认最多 `8` 个。

每个角色当前会给：

- 名字
- 叙事定位
- 所属势力
- 职业与职业补充
- 年龄
- 简介
- 性格
- 目标
- 冲突
- 秘密
- 备注

### 势力上下文

按相关性排序后截断，当前默认最多 `6` 个。

每个势力当前会给：

- 名字
- 类型
- 领袖
- 立场
- 目标
- 摘要
- 详情摘要

### 世界观上下文

按相关性排序后截断，当前默认最多 `8` 条。

每条设定当前会给：

- 类型
- 标题
- 摘要
- 详情摘要
- 标签

### 人物关系上下文

按相关性排序后截断，当前默认最多 `8` 条。

每条关系当前会给：

- 发起角色
- 指向角色
- 关系类型
- 摘要
- 细节摘要
- 强度
- 可见性

### 人物-势力关系上下文

按相关性排序后截断，当前默认最多 `8` 条。

每条关系当前会给：

- 角色
- 势力
- 关系类型
- 头衔
- 立场
- 摘要
- 细节摘要
- 是否主归属

### 物品上下文

按相关性排序后截断，当前默认最多 `6` 项，可由环境变量控制：

- `NOVEL_CONTEXT_MAX_ITEM_ITEMS`

优先显示“当前活跃持有关系”，否则回退到物品主档案。

活跃持有关系模式下会给：

- 物品名
- 持有人
- 持有方式
- 说明

物品主档案模式下会给：

- 物品名
- 分类
- 稀有度
- 状态
- 描述
- 来源

### 钩子上下文

分 3 组展示：

- 已绑定到本章的钩子
- 本章目标回收钩子
- 当前仍活跃的钩子

每类会尽量给：

- 标题
- 类型
- 当前状态
- 本章动作
- 计划说明
- 实际落地
- 优先级
- 起始章节
- 目标章节
- 摘要
- 设钩说明
- 回收说明

### 最近正式状态

这里是“上一批已经 approve 生效的信息”。

当前会给：

- 最近 N 章正式状态摘要
- 最近章节快照的原始 JSON 预览
- 最近人物状态
- 最近势力状态
- 最近钩子状态

其中：

- 最近章节正式状态摘要默认最多 `5` 条，由 `NOVEL_CONTEXT_MAX_CHAPTER_SNAPSHOT_ITEMS` 控制
- 最近人物 / 势力 / 钩子状态默认最多 `8` 条，由 `NOVEL_CONTEXT_MAX_STATE_ITEMS` 控制

单字段文本长度也支持环境变量控制，当前默认值均为 `512`：

- 普通文本字段：`NOVEL_CONTEXT_TEXT_FIELD_MAX_LENGTH`
- 较长文本字段：`NOVEL_CONTEXT_LONG_TEXT_FIELD_MAX_LENGTH`
- 最近章节快照原始 JSON 预览：`NOVEL_CONTEXT_SNAPSHOT_RAW_PAYLOAD_PREVIEW_LENGTH`

最近人物状态会给：

- 角色名
- 来源章节
- 状态摘要
- 地点
- 目标
- 外显印象
- 内在状态

最近势力状态会给：

- 势力名
- 来源章节
- 状态摘要
- 权力变化
- 对外关系

最近钩子状态会给：

- 钩子名
- 来源章节
- 进度状态
- 进度说明

## 3. Draft 链路输入

`draft write` 和 `plan` 一样，也有 `systemPrompt + prompt + contextText` 三层。

不同点是：`draft` 的任务输入比 `plan` 更“执行向”。

## 3.1 systemPrompt

`draft write` 的系统提示主要强调：

- 你是中文网络小说写作助手
- 要基于章节规划与上下文生成可直接阅读的草稿
- 不能泄露提示词、上下文、模型身份
- 不能输出元信息
- 正文优先保证：
  - 情节推进
  - 人物一致性
  - 钩子延续
  - 中文可读性

## 3.2 prompt

`draft write` 的主 prompt 当前包括：

- 项目名
- 章节标题
- 章节摘要
- 作者意图
- 额外指令
- 当前有效 plan 全文
- 本章已绑定钩子
- 当前关键物品

其中：

### 本章已绑定钩子

每条现在会给：

- 钩子标题
- 本章动作
- 钩子类型
- 钩子状态
- 计划说明
- 已落地说明

### 当前关键物品

每条现在会给：

- 物品名
- 持有人
- 持有方式
- 说明

## 3.3 contextText

`draft` 的 `contextText` 与 `plan` 共用同一份统一上下文格式。

也就是说，`draft` 实际拿到的是：

- `plan` 的全文
- 作者意图
- 额外指令
- 钩子/物品专门摘要
- 再加上一整份统一章节上下文

所以 `draft` 比 `plan` 明显更重，也更依赖上下文质量。

## 4. 这些内容是“全传”吗

不是。

当前策略是：

- 先从数据库中取出一批结构化信息
- 数据库读取阶段会先受 `NOVEL_CONTEXT_SQL_QUERY_LIMIT` 限制候选池大小，当前默认 `512`
- 再按“与本章相关性”排序
- 再按环境变量上限截断
- 再把单条文本字段做摘要截断

所以现在属于：

- 不是全库直塞
- 不是所有字段原样透传
- 而是“相关信息优先 + 有上限的摘要输入”

## 5. 当前截断与控长规则

默认上限来自环境变量：

- `NOVEL_CONTEXT_MAX_CHARACTER_ITEMS=8`
- `NOVEL_CONTEXT_MAX_FACTION_ITEMS=6`
- `NOVEL_CONTEXT_MAX_LORE_ITEMS=8`
- `NOVEL_CONTEXT_MAX_RELATION_ITEMS=8`
- `NOVEL_CONTEXT_MAX_CHARACTER_FACTION_ITEMS=8`
- `NOVEL_CONTEXT_MAX_ITEM_ITEMS=6`
- `NOVEL_CONTEXT_SQL_QUERY_LIMIT=512`
- `NOVEL_CONTEXT_MAX_CHAPTER_SNAPSHOT_ITEMS=5`
- `NOVEL_CONTEXT_MAX_HOOK_ITEMS=8`
- `NOVEL_CONTEXT_MAX_STATE_ITEMS=8`
- `NOVEL_CONTEXT_TEXT_FIELD_MAX_LENGTH=512`
- `NOVEL_CONTEXT_LONG_TEXT_FIELD_MAX_LENGTH=512`
- `NOVEL_CONTEXT_SNAPSHOT_RAW_PAYLOAD_PREVIEW_LENGTH=512`

单字段文本目前还会做长度裁剪：

- 普通文本字段：默认 `512` 字符
- 较长字段：默认 `512` 字符
- 最近章节快照原始 JSON 预览：默认 `512` 字符

## 6. 当前仍然没有直接传给 AI 的

下面这些信息仍然不是当前主链路输入的一部分，或者没有完整透传：

- 历史 `generation_runs`
- 历史 plan / 历史 draft 的全文
- 其他章节的完整正文
- 全量历史快照明细
- review 历史报告全文
- 全量人物 / 势力 / 钩子 / 关系（只会传相关性更高的一部分）

## 7. 当前设计的实际结论

可以把当前输入理解成：

- `plan`：吃“章节任务 + 精炼后的统一项目上下文”
- `draft`：吃“章节任务 + plan 全文 + 钩子/物品专门摘要 + 同一份统一上下文”

也就是说，现在已经不是“只给名字和标题”的轻量输入了，而是“重要字段增强版”的中高密度输入。

但它仍然不是“把数据库全部重要信息原样塞给模型”，而是“对本章最有用的一部分信息做结构化摘要后再传”。
